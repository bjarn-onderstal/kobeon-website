Build a visually stunning, highly interactive enterprise B2B website for Kobeon — a Dutch Mendix + AI software partner. The benchmark is capisoft.nl: a site where almost every section contains a LIVING, MOVING product demo — not static text-on-color. The single most important instruction in this entire brief:

>>> Wherever Capisoft would embed a looping product/demo video, YOU MUST BUILD A SELF-ANIMATING FAKE PRODUCT UI IN CODE. <<<

These are small, looping, auto-playing interface mockups — a dashboard whose charts draw themselves, a workflow whose tasks tick off one by one, an AI agent that types and fills a form, integration nodes that pulse data along connecting lines. They run on a loop like a video, with no user action required. This is what makes Capisoft feel alive and expensive. A site that is just bold headings on alternating purple/white backgrounds is BORING and is exactly what we are trying to avoid. If a section is only text + a color, you have done it wrong — add motion, a mock UI, floating data cards, or a live visual.

## THE CORE PATTERN — "MOCK PRODUCT DEMO" COMPONENTS (read first)
================================================================

Build a reusable set of animated, looping "fake software" components. Each is a stylized browser/app window (rounded corners, subtle top bar with 3 dots, soft shadow, glass or dark surface) containing UI that animates on an infinite loop using Framer Motion / CSS keyframes. They should feel like you are watching a screen recording of real software.

Build these specific mock demos and reuse them across the site:

1. **MockDashboard** — a stylized analytics dashboard. Bars grow from 0, a line chart path draws itself (stroke-dashoffset animation), 2–3 KPI tiles count up, a small donut fills. Loops every ~6s. Used for "Portalen & dashboards" and the hero.

2. **MockWorkflow** — a vertical/horizontal process flow of 4–5 task chips ("Order binnen", "Validatie", "Verwerking", "Factuur", "Klaar"). One by one they switch from grey → teal with a checkmark, a progress bar fills, then it resets and loops. Represents Procesautomatisering.

3. **MockAgent** — a chat/agent panel. An AI message types out character by character ("Ik verwerk 14 aanvragen..."), then shows "tool calls" appearing as little pills (Dynamics 365 ✓, document gecontroleerd ✓, e-mail verstuurd ✓), then a form auto-fills. Loops. Represents Agentic AI.

4. **MockTraining** — an AI model training view. A loss curve animates downward, an accuracy number ticks up toward 94%, a row of data records streams in. Represents AI Development.

5. **MockIntegration** — 4–5 system nodes positioned around a central Mendix node (HOMEPAGE = enterprise: Microsoft Dynamics 365, SAP, Azure, REST API, OpenAI). Animated dots travel along connecting lines between the outer nodes and the center, pulsing. Represents Systeemintegratie. (Op sectorpagina's mogen branche-specifieke nodes — Agriware/Infor/Metacom voor horticultuur, AFAS/Nmbrs voor staffing.)

6. **MockPortal** — a clean customer/employee portal: sidebar nav, a table whose rows fade in, a status badge flipping from "In behandeling" → "Goedgekeurd". Represents Portalen & legacy modernisering.

**Floating metric cards (Capisoft signature):** over/around several of these mock UIs, float 1–2 small glass cards with a live-counting number, e.g. "Tijdsbesparing +40 u/week ↑", "Besparing €100K/jaar", "Doorlooptijd −62%". They drift gently (subtle y-axis float, 4s ease-in-out loop) and the number counts up when in view. Use these on the hero and on service rows — they are a huge part of the premium feel.

Performance: these are CSS/SVG/Framer Motion only — no real video files, no heavy libs. Pause animations when off-screen (Intersection Observer) and respect prefers-reduced-motion.

Q1. Waarom een agentic AI-platform en niet 'gewone' AI zoals ChatGPT?
A1. Een losse AI-tool beantwoordt vragen; een agentic platform voert werk uit. Het verschil zit in regie en context. Een standaard chatbot heeft geen veilige toegang tot uw systemen, kent uw processen niet en handelt niets zelfstandig af. Een agentic oplossing op Mendix krijgt — binnen strikte governance — toegang tot uw data en applicaties (AFAS, SAP, Agriware), volgt uw bedrijfsregels en voert complete processen end-to-end uit, met een mens die goedkeurt waar dat nodig is. Het resultaat is geen 'slim antwoord', maar een taak die af is. En omdat alles binnen één auditeerbare laag draait, komt u uit de pilotfase waar losse AI-experimenten vaak in blijven steken.

Q2. Waarom Mendix en niet alles custom (high-code) laten bouwen?
A2. High-code geeft maximale vrijheid, maar tegen een prijs: lange doorlooptijden, hoge kosten en een codebase die afhankelijk is van schaarse specialisten. Mendix levert dezelfde enterprise-kwaliteit — security, schaalbaarheid, integraties — maar dan modelgedreven, waardoor wij in weken bouwen wat custom-teams in maanden doen. U houdt volledige flexibiliteit (eigen code kan altijd worden toegevoegd waar dat zinvol is), zonder volledig afhankelijk te worden van één ontwikkelaar of een dichtgetimmerde codebase. Kort gezegd: de snelheid van low-code, de degelijkheid van enterprise software, en een MVP in zes weken in plaats van een traject van een jaar.

Q3. Is low-code wel veilig en schaalbaar genoeg voor bedrijfskritische toepassingen?
A3. Ja. Mendix staat negen jaar op rij als Leider in de Gartner Magic Quadrant voor enterprise low-code en draait bij organisaties als Rabobank, ABN AMRO, PostNL, Schiphol en de gemeente Rotterdam. Security, schaalbaarheid en performance zijn ingebouwd — niet iets dat achteraf moet worden 'aangezet'. Applicaties schalen van tien tot honderdduizenden gebruikers, draaien cloud-native of on-prem, en voldoen aan ISO 27001, AVG en BIO. Low-code is hier geen 'lichte' variant, maar de manier waarop enterprises hun bedrijfskritische software bouwen.

Q4. Onze data is gevoelig. Waar draait dit en welke AI-modellen worden gebruikt?
A4. U houdt de regie over modellen, data en infrastructuur. We koppelen het AI-model dat bij u past — een toonaangevend model, een open-source model of een eigen model binnen uw eigen omgeving — en kunnen daar vrij tussen wisselen. AI-agenten worden gegrond in úw data zonder die te verplaatsen of te vervangen: uw bestaande systemen blijven het bronsysteem. En u bepaalt waar het draait: publieke cloud, private cloud, on-prem of volledig afgeschermd (air-gapped) als regelgeving dat eist. Voor sectoren met strenge eisen — semi-publiek, onderwijs, kinderopvang — is dit precies waarom Mendix het juiste fundament is.

Q5. Hoe voorkomen we dat onze AI in een eindeloze pilot blijft hangen?
A5. Daar is onze aanpak op gebouwd. Veel AI-initiatieven stranden omdat een losse proof-of-concept nooit veilig, geïntegreerd of beheersbaar genoeg wordt voor productie. Wij bouwen vanaf dag één op een productieklaar platform met governance, integraties en security ingebouwd. Binnen zes weken staat er een werkende MVP in uw eigen omgeving — geen demo, maar iets dat u echt gebruikt. Daarna breiden we uit in cycli van 90 dagen, zodat de oplossing meegroeit in plaats van stil te vallen.

Q6. We hebben al systemen (AFAS, SAP, Agriware, Nmbrs). Moeten die eruit?
A6. Nee. Wij bouwen bovenop en tussen wat u al heeft. Mendix is ontworpen om te integreren — via standaard connectoren en API's praten uw bestaande systemen met elkaar en met nieuwe applicaties. Vaak is de grootste winst juist het verbinden van silo's die nu niet samenwerken, zonder dat u een werkend systeem hoeft te vervangen. Legacy moderniseren we gefaseerd, nooit met een big bang.

Q7. Wat is het verschil tussen Kobeon en Mendix zelf?
A7. Mendix is het platform; Kobeon is de partner die het voor u inzet. Mendix levert de motor — security, schaalbaarheid, AI- en governance-bouwstenen, gedragen door Siemens. Wij vertalen uw processen naar werkende software op die motor: van Discovery en ontwerp tot bouw, implementatie en doorontwikkeling. U koopt geen platform om het vervolgens zelf uit te zoeken; u krijgt een team dat de oplossing bouwt, oplevert en met u laat meegroeien.

Q8. Vervangt AI onze mensen?
A8. Nee — de kracht zit in de combinatie. We orkestreren mensen, AI-agenten en systemen in één werkstroom: routinematig en cognitief werk gaat naar de agent, beslissingen en uitzonderingen blijven bij uw mensen. Uw team houdt de regie en wint tijd terug voor het werk dat er echt toe doet.

Q9. Wat kost het en hoe snel zien we resultaat?
A9. Elke samenwerking begint met een gratis Discovery-sessie van 60 minuten waarin we uw situatie en het besparingspotentieel concreet maken. Daarna werken we naar een werkende MVP in vier tot zes weken, met een transparante scope en planning. U investeert dus eerst in een klein, concreet resultaat — niet in een open-eind traject van een jaar.

## v3.4 ADDITIONS — SEO-FUNDAMENT + STERKERE CLAIMS (verwerk in elke pagina)
================================================================
================================================================

Twee dingen die in elke pagina van de build moeten landen: (A) een degelijk SEO-fundament, en (B) scherpere, autoritaire claims zodat Kobeon zichzelf niet langer tekortdoet. Houd al het andere intact.

----------------------------------------------------------------
### A — SEO-FUNDAMENT (bouw dit standaard in)
----------------------------------------------------------------
**Indexeerbaarheid (kritiek):** de site moet door zoekmachines te crawlen zijn. Als dit een React/Vite-app is, gebruik **SSR, SSG of prerendering** (bv. via een prerender-stap of een framework dat statische HTML genereert per route). Pure client-side routes zonder pre-render tellen niet voor SEO — elke pagina hieronder moet een echte, indexeerbare URL met server-gerenderde inhoud zijn.

**Per pagina verplicht:**
- Unieke `<title>` (50–60 tekens) met primair keyword + merk. Unieke `<meta name="description">` (150–160 tekens) met keyword + bewijs + CTA.
- Exact één semantische `<h1>` per pagina, met het primaire keyword — **niet** opgeknipt in losse woord-spans (animeer met CSS op intacte tekst, zodat de DOM één schone H1 houdt).
- Logische H2/H3-hiërarchie met secundaire keywords waar natuurlijk.
- Open Graph + Twitter-card tags (title, description, image) voor nette social-previews.
- Beschrijvende `alt`-teksten op álle betekenisvolle afbeeldingen en logo's (met natuurlijke keywords, bv. `alt="ABN AMRO — klant van Kobeon"`).
- Canonical tag (self-referencing) per pagina.

**Title/meta per pagina (gebruik deze, pas aan waar nodig):**
- Home — `Mendix Partner Nederland | Enterprise software & AI op maat – Kobeon` / `Kobeon bouwt enterprise-software op Mendix — 9 jaar Gartner Leader. Van procesautomatisering tot AI op maat. Werkende MVP in 6 weken. Plan een gratis Discovery-sessie.`
- Diensten — `Mendix & AI diensten — procesautomatisering tot agentic AI | Kobeon`
- Elke dienst-pagina — `[Dienst] op Mendix | Kobeon` (bv. `Procesautomatisering op Mendix | Kobeon`)
- Elke sector-pagina — `Software voor [sector] op Mendix | Kobeon`
- Projecten — `Klantcases — Mendix-software in de praktijk | Kobeon`
- Werkwijze — `Onze werkwijze — van idee naar Mendix-software in 6 weken | Kobeon`
- Werken bij — `Werken bij Kobeon — Mendix Developer & Consultant vacatures`
- Quickscan — `Gratis digitale Quickscan | Kobeon`

**Structured data (JSON-LD):**
- `Organization` (naam Kobeon, logo, adres Noordzeelaan 50, 8017 JW Zwolle, telefoon, e-mail info@kobeon.nl, sameAs social) — sitebreed.
- `FAQPage` op de FAQ-sectie (homepage + sectorpagina's) — kan FAQ-rich-results opleveren.

- Lazy-load afbeeldingen/video; comprimeer de Nano-Banana-assets en screenshots (WebP, redelijke bestandsgroottes); reserveer afmetingen om CLS te voorkomen.

## v3.5 ADDITIONS — DIENST-TEGELS MET LOOPENDE ACHTERGRONDVIDEO (Capisoft-stijl)
================================================================
================================================================

Doel: de dienstensectie hetzelfde "levende" gevoel geven als capisoft.nl, waar achter elke dienst-tegel een loopende video draait. Dit vervangt/verrijkt de statische achtergrond-images uit v3.3. Houd al het andere intact.

----------------------------------------------------------------
### WAT CAPISOFT TECHNISCH DOET (en wij nabouwen)
----------------------------------------------------------------
Elke dienst-tegel = een container met: (1) een full-bleed loopende `<video>` als achtergrond, (2) een donkere gradient-overlay voor leesbaarheid, (3) de tekst-content erbovenop. De video draait `autoplay loop muted playsinline` met `object-fit: cover` en een **poster-image** als fallback, in twee formaten (`.mp4` + `.webm`). De grid komt met een subtiele 3D-diepte binnen (cube-stijl).

----------------------------------------------------------------
### COMPONENT — `<ServiceTile>` (met video-achtergrond)
----------------------------------------------------------------
Structuur (per tegel):
- **Container:** `position: relative; overflow: hidden; border-radius: 16–20px;` met een vaste verhouding (bento-tegels) en een zachte schaduw.
- **Achtergrondvideo (laag 1):** `<video autoplay loop muted playsinline preload="none" poster="/public/visuals/svc-<id>.jpg">` met bronnen `svc-<id>.webm` én `svc-<id>.mp4`. CSS: `position:absolute; inset:0; width/height:100%; object-fit:cover;`. (muted + playsinline zijn VERPLICHT, anders weigert de browser autoplay — zeker op iOS.)
- **Overlay (laag 2):** donkere gradient van onder (#0D0D1A ~90%) naar boven (transparant), zodat titel/tekst altijd leesbaar zijn. Op hover iets donkerder.
- **Content (laag 3):** `position:relative; z-index:2;` — klein teal icoon, titel (wit, bold), 1 regel beschrijving (op grote tegels), en "Meer info →".
- **Hover/3D:** subtiele 3D-tilt (max ~6°) + lichte scale (1.02); video speelt door, overlay donkert, "→" schuift in. Entrance: tegels komen met staggered diepte/scale in (cube-gevoel), 60ms per tegel.

**Fallback-keten (belangrijk, zelfde filosofie als de rest van de site):**
1. Video aanwezig (`svc-<id>.webm/.mp4`) → toon loopende video.
2. Geen video, wel still (`svc-<id>.jpg`, de Nano Banana-render) → toon still met trage Ken Burns + de bestaande hover-zoom.
3. Geen assets → merk-gradient + (op de feature-tegels) een gecodeerde mock-animatie.

----------------------------------------------------------------
### PERFORMANCE (cruciaal — niet 9 video's tegelijk autoplayen)
----------------------------------------------------------------
Negen autoplay-video's tegelijk maakt de pagina traag en leegt accu's. Daarom:
- **Hybride aanpak (aanbevolen):** zet video alleen op de **2–3 grote "feature"-tegels** (bv. Procesautomatisering, Agentic AI, Portalen). De overige (kleinere) tegels gebruiken de gegronde Nano Banana-**still** met subtiele Ken Burns/hover-zoom. Visueel net zo rijk, veel lichter.
- **Lazy + pauzeren:** laad/start een video pas wanneer de tegel in beeld komt (IntersectionObserver) en pauzeer 'm zodra hij uit beeld is. `preload="none"`.
- **Klein houden:** elke clip 6–10s, naadloze loop, gecomprimeerd (mp4 H.264 + webm VP9/AV1), richt op < ~1,5 MB per clip. Lever altijd een geoptimaliseerde poster (`.jpg`) mee.
- **`prefers-reduced-motion`:** toon alleen de poster (geen video, geen Ken Burns).
- **Mobiel:** standaard alleen posters; eventueel 1 video op de hoofd-feature-tegel.

----------------------------------------------------------------
### HOE MAAK JE DE CLIPS? (3 routes, van "echtst" naar "snelst")
----------------------------------------------------------------
Asset-namen: `/public/visuals/svc-<id>.mp4` + `svc-<id>.webm` + poster `svc-<id>.jpg`.

1. **Schermopname van échte klantsoftware (meest geloofwaardig — sluit aan op je wens voor "echter").** Neem 6–10s van de echte app op (een dashboard dat laadt, een workflow die afvinkt, een portaal dat scrollt), anonimiseer gevoelige data (zie de Screenshot-brief §7), exporteer naar mp4+webm. Dit is wat Capisoft in feite doet en oogt het minst "nep".
2. **De gegronde Nano Banana-still animeren tot een korte loop.** Voer de realistische still (uit de Nano Banana-brief) als startframe in Veo 3 / Runway / Kling met een rustige bewegingsprompt ("very gentle camera drift over a clean workspace, soft daylight shifting, calm, seamless loop, no glowing effects"). Exporteer kort + klein.
3. **Geen videobestand — live in code.** Laat de tegel de still tonen met Ken Burns + de gecodeerde mock-animatie eroverheen. Geen extra assets, lichtst van allemaal. Prima voor de kleinere tegels.

Advies: route 1 voor je 2–3 sterkste diensten (echte software!), route 3 voor de rest. Zo krijg je de Capisoft-levendigheid zonder de hele pagina vol zware video's te zetten.

De diepere 9-diensten (met de video/bento-tegels uit v3.5) mag blijven, maar zet die dan op een lichtere sectie-achtergrond zodat het geen tweede zwart blok wordt — of verплaats 'm naar /diensten.

## v3.11 — LICHTE VARIANTEN VAN ALLE MOCKUPS & DEVICE-FRAMES
================================================================
================================================================

Omdat de meeste secties nu licht zijn (v3.10), moeten de mock-demo's en device-frames een **lichte variant** krijgen — anders blijven het donkere vlakken op een lichte pagina. Geef elk mock-component en frame een `theme`-prop en laat het standaard de sectie-achtergrond volgen.

----------------------------------------------------------------
### THEME-PROP (op alle visuele componenten)
----------------------------------------------------------------
Voeg `theme="light" | "dark"` toe aan: MockDashboard, MockWorkflow, MockAgent, MockTraining, MockIntegration, MockPortal, DeviceFrame, ScreenShowcase, en de zwevende metric-kaartjes.
- **Standaard = light.** Op lichte secties (de meeste) gebruik je de lichte variant; op de paar donkere accentsecties (hero #2A0870, stats, sectoren, CTA) gebruik je de donkere variant.
- Animaties, timing en gedrag blijven identiek — alleen de kleuren/oppervlakken wisselen.

----------------------------------------------------------------
### LICHT PALET (voor de mock-UI's)
----------------------------------------------------------------
- Oppervlak/venster: wit #FFFFFF op een heel licht-grijs canvas #F7F7FB.
- Randen/scheidingslijnen: #E6E6EF (subtiel), geen harde zwarte lijnen.
- Tekst: #161616 (primair), #6B6B7B (secundair) — niet wit.
- Accenten/data: merkpaars #3A0CA3 en teal #13A6A6 voor grafieklijnen, balken, actieve states; geel #FFB703 alleen voor één highlight.
- Diepte: zachte, lichte schaduwen (bv. 0 10px 30px rgba(20,20,40,0.08)) i.p.v. gloed. **Geen neon/glow** op licht.
- Browser/topbar van het frame: licht-grijs met de 3 stippen; adresbalk-pill in #F0F0F5.

----------------------------------------------------------------
### PER COMPONENT — wat verandert in de lichte variant
----------------------------------------------------------------
- **MockDashboard** — witte kaart, lichtgrijze gridlijnen, grafieklijn in paars→teal-gradient, KPI-tegels met donkere cijfers en een klein gekleurd deltapijltje. Dondonts/bars in merkkleuren op wit.
- **MockWorkflow** — taak-chips in licht-grijs (#F1F1F6) die naar teal met witte check "flippen"; voortgangsbalk teal op licht spoor.
- **MockAgent** — wit chatpaneel, gebruikersbubbels licht-grijs, agent-bubbel licht-paars (#EDE7F6); tool-pills met donkere tekst en een teal vinkje; de "mens keurt goed"-toggle in teal.
- **MockTraining** — witte kaart, dalende loss-curve in paars, oplopend accuraat-getal in donkere tekst met teal accent; datarijen met subtiele scheidingslijnen.
- **MockIntegration** — centrale Mendix-node + omliggende nodes als lichte kaartjes met logo/lettermerk; verbindingslijnen licht-grijs met teal datapunten die pulseren (geen gloeiende neonlijnen).
- **MockPortal** — lichte sidebar, witte tabel met rijen die infaden, statusbadge die van grijs "In behandeling" naar groen "Goedgekeurd" omklapt.
- **Zwevende metric-kaartjes** — wit/licht glas (rgba(255,255,255,0.85) + lichte rand + zachte schaduw) met donker cijfer en gekleurd label, i.p.v. donker glas. Tellen nog steeds op.
- **DeviceFrame** — licht variant: lichtgrijze/zilveren bezel, witte topbar. (Donkere variant blijft voor gebruik op de donkere accentsecties.)
- **ScreenShowcase** — de overlay-gradient onder echte screenshots wordt licht (van wit/transparant) op lichte secties; gebruik de donkere overlay alleen op donkere secties.

----------------------------------------------------------------
### HERO (donkere sectie) — combineer
----------------------------------------------------------------
De hero blijft diep paars. Daar werkt een **lichte device-mockup/ScreenShowcase** juist heel goed: een wit dashboard-venster met zachte schaduw "zweeft" op het paars en springt eruit. Gebruik dus op de hero de lichte mock-variant in een (licht of donker) frame — niet een donkere mock op donkere achtergrond.

----------------------------------------------------------------
### NANO BANANA SVC-BEELDEN — ook lichtere set
----------------------------------------------------------------
De dienst-achtergronden (Nano Banana) krijgen nu lichte secties, dus lever ze **lichter/luchtiger** aan: heldere werkplekken met daglicht, witte/lichte oppervlakken, merkkleur alleen als zachte tint. Voeg aan het stijl-anker toe: *"bright, airy, light background, lots of soft daylight, predominantly white/light surfaces, subtle purple/teal tint only as a light accent."* (Zo passen ze bij de lichtere bento/secties i.p.v. donkere beelden op licht.)

----------------------------------------------------------------
### TOEPASSEN
----------------------------------------------------------------
Bouw beide varianten één keer (theme-prop) en laat elke sectie de juiste kiezen op basis van zijn achtergrond. Lichte secties = lichte mocks met zachte schaduwen en merk-accenten; de paar donkere accentsecties = de bestaande donkere variant. Doel: dezelfde levendigheid, maar luchtig en premium — niet "duister".
